# T1 示例：结构化一个模糊问题

## 场景

你在评估是否采用新的架构方案，但问题本身太模糊，无法得到可靠答案。

## 输入

调用 `t1_protocol`，传入：

```
评估从单体应用迁移到微服务的可行性
```

## 输出

```
[L1 事实]
1. 当前应用是单体架构，代码量约 50 万行
2. 团队有 5 名开发人员
3. 部署周期为两周一次
4. 当前技术栈为 Python/Django

[L2 假设]
1. 微服务能提高部署频率（需验证）
2. 团队能维护分布式系统（需验证）
3. 业务逻辑可以清晰拆分（需验证）

[L3 假说]
1. 微服务降低单次部署风险
   ├─ 证据：行业案例间接支持
   └─ 争议：复杂度从部署转移到运维
2. 迁移不影响现有业务
   ├─ 证据：纯理论推演
   └─ 争议：增量迁移 vs 重写

[L4 未知]
1. 未来业务增长路径
2. 团队分布式系统学习曲线

[核心问题]
在 5 人团队和 Python/Django 的约束下，
微服务的部署灵活性是否值得分布式复杂度代价？
```

## 价值

- 把模糊的"好不好"问题变成可讨论的结构化问题
- 区分已知事实和待验证假设
- 暴露需要深入调研的关键争议点
